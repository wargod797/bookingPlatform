3.3 Booking Service Flow

  1. Receive showId and requested seat numbers.
  2. Load the Show from the repository.
  3. Load all requested seats for that show.
  4. Validate that all seats exist, are unique, and are not already booked.
  5. Mark seats as booked inside a transaction.
  6. Compute price using location-based third-ticket logic and afternoon discount logic.
  7. Persist the final Booking with timestamp and seat relations.

5. Design Decisions and Non-Functional View

5.1 Transactional Decisions

  • BookingService#bookTickets runs in a transaction to keep seat marking and booking persistence consistent.
  • TheatreService#onboardTheatre and ShowService seat allocation are transactional to keep parent-child persistence aligned.
  • Validation is performed before final save to reduce partial writes.

5.2 Cache Design

  • Ehcache via JCache is used for lightweight in-process read caching.
  • Current code caches movie lookup and keeps Redis as an optional future scale-out path.
  • This choice fits the current single-service deployment model from the exercise implementation.

5.3 Existing Theatre IT Integration

  • The partner APIs act as a simple façade for theatre onboarding, show publishing, and seat inventory submission.
  • In an extended design, these APIs can be backed by integration adapters for partner PMS or ERP systems.
  • For higher throughput, partner updates can be accepted synchronously and published to downstream topics asynchronously.

5.4 Scalability and Availability Direction

  • Run multiple stateless Spring Boot instances behind a load balancer.
  • Move from local cache to Redis or another distributed cache if horizontal scaling requires shared cache state.
  • Use PostgreSQL high availability, read replicas, and multi-AZ deployment for 99.99% availability goals.
  • Shard or partition by city/country when traffic or catalog volume grows significantly.

5.5 Security and Compliance

  • Input validation and exception handling are in place for the current scope, but authentication and authorization should be added for production.
  • Recommended next steps: JWT or OAuth2 for partner/customer identity, rate limiting, audit logs, secrets management, and dependency scanning.
  • Payment integration should isolate PCI-sensitive boundaries to a dedicated payment service and gateway redirect/token model.

6. Test Scenarios and Curl Commands

These examples use curl.exe with PowerShell line continuation. If your database already has data, replace movieId, theatreId, showId, and bookingId with the actual values returned by your environment.

6.1 Base Setup for an Offer Location

This sequence creates an eligible location for the third-ticket offer: Mumbai + PVR Andheri.

# 1. Onboard theatre partner
curl.exe -X POST "http://localhost:8080/partners/theatres" `
  -H "Content-Type: application/json" `
  -d '{"theatreName":"PVR Andheri","cityName":"Mumbai"}'

# 2. Create a movie
curl.exe -X POST "http://localhost:8080/movies" `
  -H "Content-Type: application/json" `
  -d '{"title":"Interstellar","genre":"Sci-Fi","language":"English"}'

# 3. Create an afternoon show in the offer location
curl.exe -X POST "http://localhost:8080/partners/shows" `
  -H "Content-Type: application/json" `
  -d '{"movieId":1,"theatreId":1,"showDate":"2026-04-04","showTime":"13:00","price":100.0}'

# 4. Allocate seat inventory
curl.exe -X POST "http://localhost:8080/partners/shows/1/seats" `
  -H "Content-Type: application/json" `
  -d '{"seatNumbers":["A1","A2","A3","A4","A5"]}'

6.2 Read Scenario: Browse Theatres Running the Show by City and Date

This validates the read scenario from the exercise.

curl.exe "http://localhost:8080/browse/shows?movieId=1&city=Mumbai&date=2026-04-04"

Expected result: a non-empty array with the configured show, theatre details, show date, show time, and price.

6.3 Write Scenario: Book Tickets for an Eligible Afternoon Offer Show

This validates both pricing rules together:

  • third-ticket 50% off because the show is in an eligible city/theatre
  • 20% afternoon discount because the show time is 13:00
curl.exe -X POST "http://localhost:8080/bookings" `
  -H "Content-Type: application/json" `
  -d '{"showId":1,"seats":["A1","A2","A3"]}'

Expected price calculation:

3 x 100 = 300
third ticket 50% off => 300 - 50 = 250
afternoon 20% off   => 250 x 0.8 = 200

Expected response field: "totalPrice": 200.0

7. Suggested Discussion Points for the Exercise

  • Why this scope: it directly demonstrates API contract, service logic, data model, design decisions, and curl-verifiable scenarios.
  • Why Ehcache now: current local single-service deployment benefits from low-latency in-process caching without extra operational complexity.
  • How to extend: add partner show update/delete, bulk booking/cancellation, payment workflow, and distributed cache if horizontal scale is required.
  • How to productionize: add authn/authz, idempotency, optimistic locking, observability, CI/CD gates, and stronger test automation.
